# Story 5.10: Testes End-to-End do Fluxo de Geração ECF

## Status

**Blocked**

---

## Story

**As a** desenvolvedor,
**I want** testes E2E cobrindo o fluxo completo de geração do Arquivo Parcial, upload do ECF Importado e merge para ECF Completo,
**so that** eu garanta que o pipeline funciona corretamente de ponta a ponta.

---

## Acceptance Criteria

1. **Teste E2E `GenerateArquivoParcialFlowTest`:**
   - Setup: cria empresa, CONTADOR, ContasParteB e LancamentosParteB (IRPJ e CSLL, meses Jan-Dez, `tipoRelacionamento` variados)
   - Executa: `POST /api/v1/ecf/generate-parcial` com `fiscalYear=2024`
   - Valida: `EcfFile.fileType = ARQUIVO_PARCIAL`, `fileStatus = DRAFT`
   - Valida: conteúdo contém M030 para cada mês com lançamentos
   - Valida: lançamentos IRPJ geraram M300/M305/M310; CSLL geraram M350/M355/M360
   - Valida: lançamento com `tipoRelacionamento = AMBOS` gerou tanto M305 quanto M310
   - Valida: `totalValor` no M300 = soma dos `valor` dos lançamentos do grupo
   - Executa: `POST /api/v1/ecf/{id}/validate`
   - Valida: `fileStatus = VALIDATED`
   - Executa: `GET /api/v1/ecf/{id}/download`
   - Valida: arquivo baixado é idêntico ao conteúdo salvo no banco

2. **Teste E2E `UploadImportedEcfFlowTest`:**
   - Setup: prepara arquivo ECF mock com bloco M válido (M001, M010, M030|A01, M300|6, M305, M310, M990)
   - Executa: `POST /api/v1/ecf/upload-importado?fiscalYear=2024` com o arquivo
   - Valida: `EcfFile.fileType = IMPORTED_ECF`, `fileStatus = DRAFT`
   - Executa: `GET /api/v1/ecf/{id}/download`
   - Valida: arquivo baixado idêntico ao uploadado (byte a byte)

3. **Teste E2E `MergeByKeyFlowTest` (validação principal do algoritmo de merge):**
   - Setup: LancamentosParteB com código `"6"` (mesmo que ECF Importado tem) e código `"99"` (não existe no Importado)
   - Setup: ECF Importado com M300|6 (valor antigo = 500,00) e M300|8 (sem equivalente no Parcial)
   - Executa: gera Arquivo Parcial (`POST /ecf/generate-parcial`)
   - Executa: upload ECF Importado (`POST /ecf/upload-importado`)
   - Executa: gera ECF Completo (`POST /ecf/generate-completo`)
   - Valida: **M300|6 no resultado usa valor do Parcial** (replace bem-sucedido)
   - Valida: **M300|8 no resultado usa valor do Importado** (preservado — sem match)
   - Valida: **M300|99 do Parcial foi adicionado** ao resultado
   - Valida: **M010 do Importado preservado** (nunca substituído)
   - Valida: **Conteúdo antes de M001 e após M990 idêntico ao Importado** (resto do ECF intacto)
   - Valida: **totalValor do M300|6 = soma dos M305 filhos** (totalizador recalculado)
   - Valida: **M990 count correto** (contagem total de linhas do bloco M)
   - Executa: `POST /ecf/{id}/validate`
   - Valida: `fileStatus = VALIDATED`
   - Executa: `PATCH /ecf/{id}/finalize`
   - Valida: `fileStatus = FINALIZED`

4. **Teste E2E `M305M310RoutingTest` (validação do roteamento de filhos):**
   - Setup: lançamentos IRPJ com os 3 tipos de `tipoRelacionamento`
   - Valida: `CONTA_PARTE_B` → apenas M305, nenhum M310
   - Valida: `CONTA_CONTABIL` → apenas M310, nenhum M305
   - Valida: `AMBOS` → M305 + M310 para o mesmo lançamento
   - Valida: `indicador` no M300 correto para cada caso (1, 2 ou 3)

5. **Teste E2E `MultiTenantIsolationTest` (isolamento de dados):**
   - Setup: duas empresas distintas com CONTADORs distintos e EcfFiles distintos
   - Valida: CONTADOR da empresa A não consegue acessar, baixar ou validar arquivos da empresa B (403)
   - Valida: listagem retorna apenas arquivos da empresa própria (nada da outra empresa)

6. Todos os testes usam TestContainers com PostgreSQL

7. Todos os testes criam contexto completo: usuário CONTADOR autenticado, empresa, header `X-Company-Id`, token JWT

8. Cobertura de código do Epic 5 deve ser ≥ 80% após as stories 5.1–5.9
