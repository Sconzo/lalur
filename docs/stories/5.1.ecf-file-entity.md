# Story 5.1: Entidade EcfFile (Três Tipos de Arquivo)

## Status

**Ready for Review**

---

## Story

**As a** desenvolvedor,
**I want** entidade EcfFile para armazenar os três tipos de arquivo ECF com seu conteúdo diretamente no banco de dados,
**so that** possamos gerar, consultar e exportar os arquivos sem depender de filesystem externo.

---

## Acceptance Criteria

1. Entidade JPA `EcfFileEntity` criada estendendo `BaseEntity` com as seguintes colunas:
   - `@Enumerated(STRING) @Column(nullable=false) EcfFileType fileType`
   - `@ManyToOne @JoinColumn(nullable=false) CompanyEntity company`
   - `@Column(nullable=false) Integer fiscalYear`
   - `@Column(nullable=false, columnDefinition="TEXT") String content`
   - `@Column String fileName`
   - `@Enumerated(STRING) @Column EcfFileStatus fileStatus`
   - `@Column(columnDefinition="TEXT") String validationErrors` (JSON array)
   - `@Column LocalDateTime generatedAt`
   - `@Column String generatedBy`
   - `@ManyToOne @JoinColumn(nullable=true) EcfFileEntity sourceImportedEcf`
   - `@ManyToOne @JoinColumn(nullable=true) EcfFileEntity sourceParcialFile`
   - `@Table(uniqueConstraints = @UniqueConstraint(columnNames = {"file_type", "company_id", "fiscal_year"}))`

2. Enum `EcfFileType` criado:
   - `ARQUIVO_PARCIAL`: bloco M gerado pelo sistema a partir dos Lançamentos da Parte B
   - `IMPORTED_ECF`: ECF completo importado pelo usuário de sistema externo
   - `COMPLETE_ECF`: ECF completo resultado do merge

3. Enum `EcfFileStatus` criado: `DRAFT`, `VALIDATED`, `ERROR`, `FINALIZED`

4. Domain model `EcfFile` criado como POJO puro em `domain/model/` com todos os campos correspondentes

5. `EcfFileRepositoryPort` criado em `application/port/out/` com os métodos:
   - `EcfFile saveOrReplace(EcfFile ecfFile)`: upsert por `(fileType, companyId, fiscalYear)`
   - `Optional<EcfFile> findById(Long id)`
   - `List<EcfFile> findByCompanyAndFiscalYear(Long companyId, Integer fiscalYear)`
   - `Optional<EcfFile> findByCompanyAndFiscalYearAndType(Long companyId, Integer fiscalYear, EcfFileType type)`

6. Adapter `EcfFileRepositoryAdapter` implementa `EcfFileRepositoryPort`; `saveOrReplace` realizado via `findByCompanyAndFiscalYearAndType` + update de campos se existir, insert se não existir

7. Migration Flyway cria a tabela `tb_ecf_file` com o unique constraint em `(file_type, company_id, fiscal_year)`

8. Mapper MapStruct `EcfFileMapper` criado para conversão `EcfFileEntity` ↔ `EcfFile`

9. Teste de integração (TestContainers) valida:
   - Salvar e recuperar por tipo e empresa (incluindo campo `content` íntegro)
   - Upsert: salvar duas vezes o mesmo `(fileType, companyId, fiscalYear)` resulta em 1 único registro com o conteúdo mais recente
   - `content` do ARQUIVO_PARCIAL é recuperado íntegro (sem truncamento)
   - Tipos distintos do mesmo ano/empresa coexistem como registros separados

### Comportamento do Upsert (saveOrReplace)

```java
// Lógica do adapter:
Optional<EcfFileEntity> existing = jpaRepo
    .findByCompanyIdAndFiscalYearAndFileTypeAndDeletedAtIsNull(
        companyId, fiscalYear, fileType);

if (existing.isPresent()) {
    mapper.updateEntity(domain, existing.get()); // atualiza campos in-place
    return mapper.toDomain(jpaRepo.save(existing.get()));
} else {
    return mapper.toDomain(jpaRepo.save(mapper.toEntity(domain)));
}
```

### Campo content (TEXT)

O PostgreSQL `TEXT` não tem limite de tamanho. Para arquivos grandes, o Hibernate carrega o campo por demanda. Avaliar uso de `@Lob` se necessário dependendo do driver, mas `columnDefinition="TEXT"` é suficiente para PostgreSQL.
